## 📖 專業術語白話對照表 (Technical Terms Glossary)

為了方便非技術背景讀者閱讀，以下將文中提及的程式術語，對照為日常生活的概念：

| 專業術語 | 英文原文 | 生活化白話比喻與說明 |
| :--- | :--- | :--- |
| **網頁狀態變數 (State)** | React State | **「儀表板上的數字」**。網頁元件自己記住的內部數據（如血量、子彈數）。一旦數值改變，網頁畫面會自動重畫更新。 |
| **元件參數傳遞 (Props)** | React Props | **「上級發放的公文」**。父級元件（如遊戲主控台）分發給子元件（如武器）的外部參數（如當前地圖類型）。 |
| **狀態監聽器 (useEffect)** | React useEffect | **「自動感應警報器」**。React 框架的一種機制，設定「當某些變數改變時，自動去執行後續特定的動作」（如：敵人出生了，就自動幫他找最近的掩體）。 |
| **視覺模型實體指標 (meshRef.current)** | Mesh Reference | **「舞台上的實體道具」**。指向遊戲中具體 3D 模型在畫面上真實被畫出來後的實體。如果是空值（`null`），代表電腦此時還沒完成這個 3D 模型的渲染。 |
| **過期閉包 (Stale Closure)** | Stale Closure | **「過期的參考相片」**。程式偵聽器在建立時拍了一張「變數快照」。如果變數改變了，但偵聽器沒有重拍，它就會一直讀取舊快照中的過期資訊。 |
| **動態記憶體暫存器 (useRef / Refs)** | React useRef | **「幕後的暫存抽屜」**。一個可以直接存取最新變數的暫存箱。在讀寫時不會重畫網頁，非常適合用來解決「過期相片」的 Bug。 |
| **狀態提升 (State Hoisting)** | State Hoisting | **「集中化管理」**。將多個子元件都需要用到的變數，從子元件內部拿出來，「提升」存放在共同的爸爸元件中統一發放，確保數據同步。 |
| **遊戲幀更新器 (useFrame)** | useFrame (Three.js) | **「心臟起搏器」**。每秒鐘執行大約 60 次（60 FPS）的計時器，用來在極短時間內不斷更新 3D 世界裡角色的位置、檢查撞牆或中彈。 |
| **邊界極限常數 (mapLimit)** | Map Boundary Limit | **「隱形圍籬限制」**。程式內部寫死的一個範圍數值，用來限制角色所能移動的最大座標範圍，防止跌落地圖邊緣外。 |
| **AABB 碰撞檢測** | Axis-Aligned Bounding Box | **「長方體包裝盒檢測」**。將複雜的 3D 模型用一個簡單的「正方盒」包起來，當角色移動時，只去計算角色有沒有碰到這個「盒子」的邊界，大幅節省運算效能。 |
| **定時器清除指令 (clearTimeout)** | clearTimeout | **「手動關閉定時鬧鐘」**。遊戲中有些動作需要時間（如換彈 1.5 秒），當動作被切換或中斷時，用來強行取消這個倒數計時鬧鐘的指令。 |

| **視線阻擋檢測 (Line of Sight)** | Line of Sight (LoS) | **「視線直視檢查」**。檢查兩個 3D 物體（如敵人和玩家）之間是否被實體遮蔽物（如水泥牆）擋住，用來判斷敵人看不看得見玩家。 |
| **三維空間向量座標 (Vector3)** | Three.js Vector3 | **「定位經緯度與高度」**。Three.js 中用來代表 3D 空間中一個點的位置 `(X, Y, Z)` 的數學物件。 |

| **網頁視窗 (Viewport)** | HTML Viewport | **「瀏覽器的看窗規格」**。用來控制網頁在不同裝置（電腦或手機）上的顯示比例與縮放限制設定。 |
| **網頁排版媒體查詢 (CSS Media Query)** | CSS Media Query | **「版面變形金剛」**。網頁排版技術，能讓網頁根據螢幕的寬高，自動切換不同的按鈕排版與大小。 |
| **觸碰事件監聽 (touchstart/touchmove)** | Touch Events | **「手指碰觸感應器」**。當手指碰到螢幕（touchstart）與在螢幕上滑動（touchmove）時，瀏覽器產生的觸控訊號。 |

| **本機快取資料庫 (LocalStorage)** | LocalStorage | **「瀏覽器的私人保險箱」**。瀏覽器內建的迷你硬碟空間，在網頁關閉後，依然能保存玩家的金幣、倉庫物品等個人資料。 |
| **雙重驗證 (2FA / TOTP)** | 2FA / TOTP | **「雙保險動態密碼」**。除了密碼外，需輸入由時間演算法每 30 秒自動更新一次的 6 位數安全碼才能登入。 |
| **畫布元件 (Canvas)** | React Three Fiber Canvas | **「3D 投影大銀幕」**。React 中用來渲染 3D 虛擬世界的專用網頁畫布容器。 |

## 📋 17 項 Bug 邏輯與修復原理全解析

### 1. 敵軍 AI 出生後集體消失問題 (Enemies Disappear Bug)
*   **遊戲中的現象**：敵人一出生就不見了，最後發現全擠在地圖中心點 `(0, 0, 0)` 的軍規木箱堆裡，像在玩躲貓貓。
*   **背後的邏輯病因**：當敵人剛被創造出來時，程式試圖去抓取敵人的 **3D 視覺實體指標 (`meshRef.current`)** 來計算離它最近的掩體。但由於電腦在渲染 3D 畫面時，"視覺實體"還沒來得及載入完成（此時為空值 `null`），**狀態監聽器 (`useEffect`)** 因為找不到這個實體，就放棄了後續的計算，直接把敵人的目標點設為預設的坐標 `(0, 0, 0)`。
*   **我是如何解決的**：不再等待視覺實體載入，而是直接拿敵人被分配好的**「出生座標數據（`data.position`）」**來計算它和周圍掩體的距離。只要有了出生座標，AI 就可以立刻計算並前進，不再迷失在中心點。

### 2. 隱形空氣牆限制問題 (Invisible Air Walls Bug)
*   **遊戲中的現象**：明明地圖邊界的圍牆已經往外移到 120 公尺了，但玩家走到大約 58 公尺時就會撞上隱形的牆壁（空氣牆）。
*   **背後的邏輯病因**：遊戲的視覺地圖（3D 圍牆模型）雖然變大了，但控制玩家移動邊界的**「邊界極限常數 (`mapLimit`)」**在程式碼中依然被寫死了舊的數值 `58` 公尺。這就好像擴建了操場圍牆，但學生的定位 GPS 警報範圍依然被限制在舊範圍內，一旦超過就強行禁止前進。
*   **我是如何解決的**：將控制邏輯中的限制常數同步修改為 `118` 公尺，讓玩家可以一路跑到視覺圍牆邊緣（120 公尺減去玩家身體半徑）。

### 3. 掩體沒有碰撞體積問題 (No Cover Collisions Bug)
*   **遊戲中的現象**：玩家可以像幽靈一樣穿過沙包、集裝箱和碉堡，沒有任何阻擋。
*   **背後的邏輯病因**：遊戲裡的障礙物此時只是一張「3D 貼圖/視覺模型」，就好像電影裡的特效牆，沒有實質的物理體積。玩家移動時，程式只計算了鍵盤輸入，沒有判定玩家是否撞上這些障礙物。
*   **我是如何解決的**：
    1.  建立一個「靜態碰撞體清單（`STATIC_COLLIDERS`）」，把地圖上所有箱子和沙包的位置與大小登記下來。
    2.  當玩家移動時，程式會同步計算玩家的圓形身體是否與這些登記在案的障礙物範圍重疊（稱為 **AABB 碰撞檢測**）。
    3.  如果發生重疊，程式會自動把玩家「推回」到障礙物的邊緣，並允許玩家沿著牆面「滑行」，從而實現流暢的物理碰撞效果。

### 4. 裝彈中途被意外取消問題 (Reload Instantly Cancelled Bug)
*   **遊戲中的現象**：按下 R 鍵要裝子彈時，畫面上的「裝彈中...」進度條閃了一下就消失了，子彈完全沒有補滿。
*   **背後的邏輯病因**：程式的防呆機制被過度觸發了。當按下 R 鍵時，遊戲狀態會變成「正在裝彈（`isReloading = true`）」。然而，原本監聽 R 鍵的**狀態監聽器 (`useEffect`)** 被設定為「一旦裝彈狀態改變，就立刻重設並清理舊計時器」。結果就是：裝彈一啟動，狀態一改變，偵聽器就誤以為要進行「清理（Cleanup）」，在一毫秒內把剛剛啟動的裝彈計時器給關閉了。
*   **我是如何解決的**：將「清理定時器（`clearTimeout`）」的時機與裝彈按鍵監聽分開，只有當玩家關閉網頁或整個遊戲關閉時，才去清理計時器，確保裝彈計時器可以完整跑完。

### 5. 手槍不能開槍/扣除步槍彈藥問題 (Pistol Firing Stale Closure Bug)
*   **遊戲中的現象**：切換成手槍後，點擊左鍵開火，手槍子彈沒有減少。切回步槍後，卻發現步槍子彈被扣掉了。如果步槍沒子彈了，手槍也跟著射不出來。
*   **背後的邏輯病因**：這在程式設計中叫作 **「過期閉包 (Stale Closure)」**。你可以把它想像成**「相片過期了」**。當遊戲一開始載入時，手槍的開火偵聽器「拍了一張快照」來記錄當前狀態。當時玩家手上拿的是步槍，所以這張相片記錄的扣彈藥對象是「步槍」。即使玩家後來切換成了手槍，因為這個偵聽器沒有重新拍照，它每次開火依然會去執行當初相片上寫的動作——扣步槍子彈。
*   **我是如何解決的**：在偵聽器中使用一個名為 `useRef` 的**「動態記憶體暫存器 (Refs)」**。這就像是不再看死相片，而是把更新彈藥的「動態視訊連結」放在那裡。每次開火時，偵聽器都會去讀取最新的狀態，確保手上拿手槍時扣手槍子彈，拿步槍時扣步槍子彈。

### 6. 敵軍子彈穿透掩體打中玩家問題 (Bullets Penetrate Covers Bug)
*   **遊戲中的現象**：玩家躲在水泥牆或沙包後，遠處的敵人射出的紅色雷射線直接穿透水泥牆打中玩家並扣血。
*   **背後的邏輯病因**：原本敵人的開火邏輯非常單純：只要玩家與敵人的距離在射程內，就直接扣玩家的血量。它並沒有考慮玩家和敵人之間是否隔著一道實體牆。
*   **我是如何解決的**：加入 **「探測射線投射 (Raycast)」** 來做 **「視線阻擋檢測 (Line of Sight)」**。在敵軍準備開槍前，程式從敵軍的眼睛位置向玩家發射一條無形的探測射線（`THREE.Raycaster`）。如果這條射線在碰到玩家之前，先撞到了地圖上的牆壁或沙包，程式就判定「視線被擋住了」，敵軍此時的射擊將無效，既不扣血也不會畫出穿牆的雷射光。

### 7. 行動端 HUD 面板遮擋與按鍵重疊問題 (Mobile HUD Overlap Bug)
*   **遊戲中的現象**：在手機上玩時，畫面一片混亂——背包面版把左手移動的虛擬搖桿擋住了；血條和子彈卡片把右手開火和跳躍鍵重疊了；右上角的暫停鈕和任務清單重疊；開火鈕甚至被手機底部的「虛擬首頁橫條」擋住，按了沒反應。
*   **背後的邏輯病因**：電腦螢幕寬度很寬，所以 **抬頭顯示器 (HUD)** 元素可以散落在兩側。但手機螢幕寬度有限，且多數人是橫屏雙手持握，如果直接套用電腦版的版面配置，按鍵就會擠在一塊。
*   **我是如何解決的**：設計了一套**「手機端網頁排版媒體查詢 (CSS Media Query)」**。當偵測到是行動裝置時：
    - 將背包移到左上角小地圖下方，空出左下角給虛擬搖桿。
    - 將血條和彈藥卡片寬度縮小一半，擠在螢幕最下方的中間區域。
    - 暫停與切槍鍵移到上方雷達兩側，避開任務說明。
    - 將右下角按鈕整體往上移高 10 像素，避開手機底部的系統手勢橫條。

### 8. 行動端視角旋轉滑動靈敏度過低問題 (Mobile Camera Rotation Insensitive)
*   **遊戲中的現象**：在手機上滑動右半邊螢幕轉動視角時，鏡頭幾乎不動，手指滑了老半天才轉一點點。
*   **背後的邏輯病因**：當玩家的手指在螢幕上移動時，程式會監聽 **觸碰事件 (touchmove)**，計算手指滑動的距離並乘上一個「靈敏度係數」來轉化為鏡頭旋轉的角度。原本的靈敏度被設定為 `0.0035`，這在手機的高解析度螢幕下顯得過小，導致轉向非常吃力。
*   **我是如何解決的**：將靈敏度參數直接加倍提升至 `0.007`，讓滑動轉向變得輕盈且靈敏。

### 9. 管理員帳號權限硬編碼與本機資料不同步問題 (Admin Lack of Persistence)
*   **遊戲中的現象**：先前如果有人註冊帳號為 `"distant star"`，就會自動變成管理員，而且如果不重新登入或重新整理網頁，權限就不會生效。
*   **背後的邏輯病因**：這是**「硬編碼 (Hardcode)」**與資料不同步的漏洞。程式直接在代碼裡寫死「如果名字叫 distant star 就是管理員」，但資料庫裡這個帳號並沒有被真的標記為管理員。這導致本地瀏覽器儲存的資料與雲端資料庫不同步，且一旦帳號更改名字，管理員權限就會遺失或被他人冒用。
*   **我是如何解決的**：
    - 移除了代碼裡的硬編碼名字判定。
    - 直接在雲端資料庫中修改該帳號的資料，加上 `"isAdmin": true`。
    - 程式在載入時，會執行一次資料移轉，把這個雲端的管理員屬性寫入 **本機快取資料庫 (LocalStorage)** 中，保證資料的持久化與安全性。

### 10. 管理員權限不當暴露與缺乏雙重驗證問題 (Admin Privilege Exposure & 2FA)
*   **遊戲中的現象**：任何人只要知道管理員的帳號名稱，就可以在一般登入框嘗試破解密碼，存在資安風險。
*   **背後的邏輯病因**：因為管理員權限極大（無限金幣、作弊指令），如果只有一般的帳號密碼防護，非常容易遭到暴力破解。
*   **我是如何解決的**：
    - **隔離一般通道**：如果有人想在常規登入框輸入管理員帳密，系統直接拒絕，並提示必須使用「金鑰登入」。
    - **雙重驗證 (2FA)**：在獨立的【金鑰登入 KEY】分頁中，使用者必須輸入：**獨家遊戲金鑰** + **個人密碼** + **動態安全驗證碼 (TOTP)**。
    - **動態驗證碼 (TOTP)**：這是一種軍規的安全技術。程式會根據當前的時間戳記，以大質數進行數學運算，產生一個每 30 秒自動更新一次的 6 位數密碼。同時在介面設計了「電子安全密碼器 Widget」與倒數條。為了防止在 30 秒交界時因為時間微小偏差導致登入失敗，還設計了前後 30 秒的時差容錯。

### 11. 敵軍 AI 撞牆/撞障礙物卡死不動問題 (Enemy AI Stuck on Walls Bug)
*   **遊戲中的現象**：在戰鬥中，部分敵軍在衝向玩家或尋找沙包時，如果撞到了障礙物的角落，就會在那裡一直跑步推牆，永遠卡在原地。
*   **背後的邏輯病因**：AI 的移動方式是直線前進。當它撞到牆時，遊戲的物理引擎會把它擋下（位移變為 0）。但因為它還沒抵達目標點，AI 的「大腦邏輯」會一直強迫它繼續往前走，導致它陷入推牆的無限循環中。
*   **我是如何解決的**：
    - **增加防卡死監控**：在 **遊戲幀更新器 (`useFrame`)** 中，每幀監控敵人的實際位移（以 **三維空間向量座標 `Vector3`** 計算）。如果發現敵人明明在「跑步」，但實際位移速度低於正常速度的 15%，就開始計時。
    - **強制轉移狀態**：一旦卡牆達 1.2 秒，程式會強行介入，清空它目前的目標，命令它重新隨機挑選別的沙包掩體，或者改為朝著玩家衝鋒，透過強迫轉向來順暢繞過障礙物。

### 12. 暫停回復自動返回初始點問題 (Pause / Resume Reset State Bug)
*   **遊戲中的現象**：在地下通道打到一半，玩家按下 ESC 鍵暫停，點擊繼續返回遊戲後，玩家的 3D 位置與通關進度被重置，回到了長廊起點。
*   **背後的邏輯病因**：當點擊暫停或回復時，遊戲 UI 需要切換。在 React 框架中，這會導致 **3D 畫布元件 (Canvas)** 整個被重新建立（掛載）。因為畫布重新建立了，所有儲存在 3D 空間裡的變數（玩家座標、已消滅的敵人數）就會像重新開機一樣被歸零。
*   **我是如何解決的**：重構了 React 的狀態保存邏輯，進行 **狀態提升 (State Hoisting)**。將玩家的 3D 坐標與地鐵出口數字（如出口 7）提升並鎖定在 3D 畫布之外的父級元件中，確保畫布重載時，這些核心數據能被正確保留並寫回 3D 空間中。

### 13. 地鐵通道飾物無碰撞體積問題 (Subway Assets Cover Collisions Bug)
*   **遊戲中的現象**：地鐵通道地圖中的自動販賣機、垃圾桶、掩體箱就像空氣一樣，玩家和敵人可以直接穿透它們。而且在「空間扭曲」事件中，當通道變彎時，碰撞邊界與視覺圖像脫節。
*   **背後的邏輯病因**：這些裝飾物的座標與大小沒有被登記到「物理碰撞地圖」中。更困難的是，當空間因為扭曲變成 S 型時，這些裝飾物的視覺模型被彎曲公式移位了，但如果碰撞邊界沒有跟著用數學公式計算移位，就會出現「看得到但撞不到，或者撞到隱形東西」的靈異現象。
*   **我是如何解決的**：
    - 建立地鐵專用的碰撞清單（`FACILITY_COLLIDERS`），將販賣機等物體登記在案。
    - 當空間扭曲事件發生時，在每一幀的碰撞判定中，將這些障礙物的座標同步套用 Sine 正弦彎曲公式進行數學投影變形，確保物理碰撞界限跟著視覺模型一起彎曲移位。

### 14. 管理員跳關後留在原地問題 (Admin Skip Wave Teleportation Bug)
*   **遊戲中的現象**：管理員使用作弊鍵「跳過波次」後，雖然順利開啟了通道門，但玩家仍停留在起點，必須自己跑 240 公尺才能走到下一關，測試起來很沒效率。
*   **背後的邏輯病因**：跳波次的邏輯只清空了敵人的生命值並開啟了閘門，但忘了「傳送玩家」。
*   **我是如何解決的**：在跳波次的邏輯中加上地理位置傳送指令：一旦在地鐵通道中使用跳過波次，自動將玩家的 3D 相機座標 Z 軸傳送到通道盡頭的撤離門附近（`z = -105` 米），免去無意義的跑圖時間。

### 15. 個人倉庫物品圖片背景與尺寸問題 (Stash Item Image Fit Bug)
*   **遊戲中的現象**：個人倉庫裡的武器（例如 AK47）和醫療包有黑底或白底的背景色，且拖曳到格子裡時，武器圖片沒有填滿它所佔據的格子，看起來非常凌亂且廉價。
*   **背後的邏輯病因**：圖片沒有進行去背處理，且 CSS 樣式中圖片的長寬比例被寫死了，沒有隨著網格系統動態拉伸。
*   **我是如何解決的**：
    - 將所有物品圖片處理為去背的 PNG 檔。
    - 修改 CSS 佈局，讓圖片的寬高能 100% 隨物品佔用的網格格數（例如武器佔用 4x2 個格子，醫療包佔用 1x1 個格子）進行自適應拉伸，使排版整齊貼合。

### 16. 點開網頁黑屏/未定義變數崩潰問題 (Weapon SelectedMap ReferenceError Bug)
*   **遊戲中的現象**：剛點開遊戲網頁，畫面一片漆黑，完全沒有加載主選單，按下鍵盤也沒反應，遊戲直接當掉。
*   **背後的邏輯病因**：這在程式中稱為 **未定義變數錯誤 (`ReferenceError`)**。因為在設計「電力中斷事件中，玩家主武器會自動開啟手電筒」的功能時，武器元件（`Weapon`）需要知道目前是什麼地圖（`selectedMap`）。但我們忘了在傳遞資料時，把這個地圖變數放進 `Weapon` 的參數清單中。React 在編譯時找不到這個變數，就判定為致命錯誤，直接讓整個網頁崩潰。
*   **我是如何解決的**：進行 **狀態提升 (State Hoisting)**。將手電筒所需的控制參數和 Reference 提升到最頂層的 `App` 元件中，再統一作為 **元件參數傳遞 (`Props`)** 傳給底下的每一個元件，消除了未定義變數的崩潰問題。

### 17. 敵軍 AI 位移碰撞未定義 activeColliders 崩潰問題 (Enemy ActiveColliders ReferenceError Bug)
*   **遊戲中的現象**：進入遊戲關卡後，畫面瞬間凍結，控制台瘋狂跳出 `ReferenceError: activeColliders is not defined` 報錯。
*   **背後的邏輯病因**：類似於 Bug 16。我們在編寫敵軍 AI 的碰撞阻擋邏輯時，直接使用了「碰撞體清單（`activeColliders`）」這個變數。然而，敵軍元件（`Enemy`）在自己內部的函數中並沒有定義這個變數，也沒有從外部傳進來。Three.js 引擎在每秒 60 幀的渲染循環中，一讀到這一行就因為找不到變數而崩潰，導致畫面卡死。
*   **我是如何解決的**：在敵軍元件的每幀渲染函數（`useFrame`更新器）頂部，動態獲取當前地圖的碰撞體清單，並套用空間扭曲公式計算變形後的座標，確保敵人在每幀移動時都能找到對應的變數進行判定。
