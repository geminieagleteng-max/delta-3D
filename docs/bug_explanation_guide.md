### 1. 敵軍 AI 出生後集體消失問題 (Enemies Disappear Bug)
*   **遊戲中的現象**：敵人一出生就不見了，最後發現全擠在地圖中心點 `(0, 0, 0)` 的軍規木箱堆裡，像在玩躲貓貓。
*   **背後的邏輯病因**：當敵人剛被創造出來時，程式試圖去抓取敵人的 **3D 視覺實體** 來計算離它最近的掩體。但由於電腦在渲染 3D 畫面時，"視覺實體"還沒來得及載入完成（此時為空值 `null`），尋路系統因為找不到這個實體，就放棄了後續的計算，直接把敵人的目標點設為預設的坐標 `(0, 0, 0)`。
*   **我是如何解決的**：不再等待視覺實體載入，而是直接拿敵人被分配好的**「出生坐標數據」**來計算它和周圍掩體的距離。只要有了出生坐標，AI 就可以立刻計算並前進，不再迷失在中心點。

### 2. 隱形空氣牆限制問題 (Invisible Air Walls Bug)
*   **遊戲中的現象**：明明地圖邊界的圍牆已經往外移到 120 公尺了，但玩家走到大約 58 公尺時就會撞上隱形的牆壁（空氣牆）。
*   **背後的邏輯病因**：遊戲的視覺地圖（3D 圍牆模型）雖然變大了，但控制玩家移動邊界的**「邏輯限制範圍（`mapLimit` 常數）」**在程式碼中依然被寫死了舊的數值 `58` 公尺。這就好像擴建了學校圍牆，但學生的定位 GPS 警報範圍依然被限制在舊範圍內，一旦超過就強行禁止前進。
*   **我是如何解決的**：將控制邏輯中的限制範圍同步修改為 `118` 公尺，讓玩家可以一路跑到視覺圍牆邊緣（120 公尺減去玩家半徑）。

### 3. 掩體沒有碰撞體積問題 (No Cover Collisions Bug)
*   **遊戲中的現象**：玩家可以像幽靈一樣穿過沙包、集裝箱和碉堡，沒有任何阻擋。
*   **背後的邏輯病因**：遊戲裡的障礙物此時只是一張「3D 貼圖/視覺模型」，就好像電影裡的特效牆，沒有實質的物理體積。玩家移動時，程式只計算了鍵盤輸入，沒有判定玩家是否撞上這些障礙物。
*   **我是如何解決的**：
    1.  建立一個「障礙物地理座標清單」，把地圖上所有箱子和沙包的位置與大小登記下來。
    2.  當玩家移動時，程式會同步計算玩家的圓形身體是否與這些登記在案的障礙物範圍重疊（稱為 AABB 碰撞檢測）。
    3.  如果發生重疊，程式會自動把玩家「推回」到障礙物的邊緣，並允許玩家沿著牆面「滑行」，從而實現流暢的物理碰撞效果。

### 4. 裝彈中途被意外取消問題 (Reload Instantly Cancelled Bug)
*   **遊戲中的現象**：按下 R 鍵要裝子彈時，畫面上的「裝彈中...」進度條閃了一下就消失了，子彈完全沒有補滿。
*   **背後的邏輯病因**：程式的防呆（Cleanup）機制被過度觸發了。在 React 網頁框架中，按下 R 鍵時，遊戲狀態會變成「正在裝彈」。然而，原本監聽 R 鍵的偵聽器被設定為「一旦裝彈狀態改變，就立刻重設並清理舊計時器」。結果就是：裝彈一啟動，狀態一改變，偵聽器就誤以為要「清理」，在一毫秒內把剛剛啟動的裝彈計時器給關閉了。
*   **我是如何解決的**：將「清理計時器」的時機與裝彈按鍵監聽分開，只有當玩家關閉網頁或整個遊戲元件被卸載時，才去清理計時器，確保裝彈計時器可以完整跑完 1.5 秒。

### 5. 手槍不能開槍/扣除步槍彈藥問題 (Pistol Firing Stale Closure Bug)
*   **遊戲中的現象**：切換成手槍後，點擊左鍵開火，手槍子彈沒有減少。切回步槍後，卻發現步槍子彈被扣掉了。如果步槍沒子彈了，手槍也跟著射不出來。
*   **背後的邏輯病因**：這在程式設計中叫作 **「過期閉包 (Stale Closure)」**。你可以把它想像成**「相片過期了」**。當遊戲一開始載入時，手槍的開火偵聽器「拍了一張快照」來記錄當前狀態。當時玩家手上拿的是步槍，所以這張相片記錄的扣彈藥對象是「步槍」。即使玩家後來切換成了手槍，因為這個偵聽器沒有重新拍照，它每次開火依然會去執行當初相片上寫的動作——扣步槍子彈。
*   **我是如何解決的**：在偵聽器中使用一個名為 `useRef` 的**「動態記憶體暫存器」**。這就像是不再看死相片，而是把更新彈藥的「動態視訊連結」放在那裡。每次開火時，偵聽器都會去讀取最新的狀態，確保手上拿手槍時扣手槍子彈，拿步槍時扣步槍子彈。

### 6. 敵軍子彈穿透掩體打中玩家問題 (Bullets Penetrate Covers Bug)
*   **遊戲中的現象**：玩家躲在水泥牆或沙包後，遠處的敵人射出的紅色雷射線直接穿透水泥牆打中玩家並扣血。
*   **背後的邏輯病因**：原本敵人的開火邏輯非常單純：只要玩家與敵人的距離在射程內，就直接扣玩家的血量。它並沒有考慮玩家和敵人之間是否隔著一道實體牆。
*   **我是如何解決的**：加入 **「視線檢測 (Line of Sight Raycast)」**。在敵軍準備開槍前，程式從敵軍的眼睛位置向玩家發射一條無形的「探測射線（類似雷射筆）」。如果這條射線在碰到玩家之前，先撞到了地圖上的牆壁或沙包，程式就判定「視線被擋住了」，敵軍此時的射射擊將無效，既不扣血也不會畫出穿牆的雷射光。

### 7. 行動端 HUD 面板遮擋與按鍵重疊問題 (Mobile HUD Overlap Bug)
*   **遊戲中的現象**：在手機上玩時，畫面一片混亂——背包面版把左手移動的虛擬搖桿擋住了；血條和子彈卡片把右手開火和跳躍鍵重疊了；右上角的暫停鈕和任務清單重疊；開火鈕甚至被手機底部的「虛擬首頁橫條」擋住，按了沒反應。
*   **背後的邏輯病因**：電腦螢幕寬度很寬，所以 UI 元素可以散落在兩側。但手機螢幕寬度有限，且多數人是橫屏雙手持握，如果直接套用電腦版的版面配置，按鍵就會擠在一塊。
*   **我是如何解決的**：設計了一套**「手機端響應式排版規則（CSS 媒體查詢）」**。當偵測到是手機橫屏時：
    - 將背包移到左上角小地圖下方，空出左下角給虛擬搖桿。
    - 將血條和彈藥卡片寬度縮小一半，擠在螢幕最下方的中間區域。
    - 暫停與切槍鍵移到上方雷達兩側，避開任務說明。
    - 將右下角按鈕整體往上移高 10 像素，避開手機底部導航條的干擾。

### 8. 行動端視角旋轉滑動靈敏度過低問題 (Mobile Camera Rotation Insensitive)
*   **遊戲中的現象**：在手機上滑動右半邊螢幕轉動視角時，鏡頭幾乎不動，手指滑了老半天才轉一點點。
*   **背後的邏輯病因**：當玩家的手指在螢幕上移動時，程式會計算手指滑動的距離，並乘上一個「靈敏度係數」來轉化為鏡頭旋轉的角度。原本的靈敏度被設定為 `0.0035`，這在手機的高解析度螢幕下顯得過小，導致轉向非常吃力。
*   **我是如何解決的**：將靈敏度參數直接加倍提升至 `0.007`，讓滑動轉向變得輕盈且靈敏。

### 9. 管理員帳號權限硬編碼與本機資料不同步問題 (Admin Lack of Persistence)
*   **遊戲中的現象**：先前如果有人註冊帳號為 `"distant star"`，就會自動變成管理員，而且如果不重新登入或重新整理網頁，權限就不會生效。
*   **背後的邏輯病因**：這是**「硬編碼判定」**與**「資料未持久化」**的漏洞。程式直接在代碼裡寫死「如果名字叫 distant star 就是管理員」，但資料庫裡這個帳號並沒有被真的標記為管理員。這導致本地瀏覽器儲存的資料與雲端資料庫不同步，且一旦帳號更改名字，管理員權限就會遺失或被他人冒用。
*   **我是如何解決的**：
    - 移除了代碼裡的硬編碼名字判定。
    - 直接在 KVdb 雲端資料庫中修改該帳號的資料，加上 `"isAdmin": true`。
    - 程式在載入時，會執行一次資料移轉，把這個雲端的管理員屬性寫入本地的 LocalStorage 快取中，保證資料的持久化與安全性。

### 10. 管理員權限不當暴露與缺乏雙重驗證問題 (Admin Privilege Exposure & 2FA)
*   **遊戲中的現象**：任何人只要知道管理員的帳號名稱，就可以在一般登入框嘗試破解密碼，存在資安風險。
*   **背後的邏輯病因**：因為管理員權限極大（無限金幣、作弊指令），如果只有一般的帳號密碼防護，非常容易遭到暴力破解。
*   **我是如何解決的**：
    - **隔離一般通道**：如果有人想在常規登入框輸入管理員帳密，系統直接拒絕，並提示必須使用「金鑰登入」。
    - **三重驗證 (2FA)**：在獨立的【金鑰登入 KEY】分頁中，使用者必須輸入：**獨家遊戲金鑰** + **個人密碼** + **動態安全驗證碼**。
    - **動態驗證碼 (TOTP)**：這是一種軍規的安全技術。程式會根據當前的時間戳記，以大質數進行數學運算，產生一個每 30 秒自動更新一次的 6 位數密碼。同時在介面設計了「電子安全密碼器 Widget」與倒數條。為了防止在 30 秒交界時因為時間微小偏差導致登入失敗，還設計了前後 30 秒的時差容錯。

### 11. 敵軍 AI 撞牆/撞障礙物卡死不動問題 (Enemy AI Stuck on Walls Bug)
*   **遊戲中的現象**：在戰鬥中，部分敵軍在衝向玩家或尋找沙包時，如果撞到了障礙物的角落，就會在那裡一直跑步推牆，永遠卡在原地。
*   **背後的邏輯病因**：AI 的移動方式是直線前進。當它撞到牆時，遊戲的物理引擎會把它擋下（位移變為 0）。但因為它還沒抵達目標點，AI 的「大腦邏輯」會一直強迫它繼續往前走，導致它陷入推牆的無限循環中。
*   **我是如何解決的**：
    - **增加防卡死監控**：每幀監控敵人的實際位移。如果發現敵人明明在「跑步」，但實際位移速度低於正常速度的 15%，就開始計時。
    - **強制轉移狀態**：一旦卡牆達 1.2 秒，程式會強行介入，清空它目前的目標，命令它重新隨機挑選別的沙包掩體，或者改為朝著玩家衝鋒，透過強迫轉向來順暢繞過障礙物。

### 12. 暫停回復自動返回初始點問題 (Pause / Resume Reset State Bug)
*   **遊戲中的現象**：在地下通道打到一半，玩家按下 ESC 鍵暫停，點擊繼續返回遊戲後，玩家的 3D 位置與通關進度被重置，回到了長廊起點。
*   **背後的邏輯病因**：當點擊暫停或回復時，遊戲 UI 需要切換。在 React 框架中，這會導致 3D 渲染畫布整個被重新載入（掛載）。因為畫布重新建立了，所有儲存在 3D 空間裡的變數（玩家座標、已消滅的敵人數）就會像重新開機一樣被歸零。
*   **我是如何解決的**：重構了 React 的狀態保存邏輯，將玩家的 3D 坐標與地鐵出口數字（如出口 7）提升並鎖定在 3D 畫布之外的父級元件中，確保畫布重載時，這些核心數據能被正確保留並寫回 3D 空間中。

### 13. 地鐵通道飾物無碰撞體積問題 (Subway Assets Cover Collisions Bug)
*   **遊戲中的現象**：地鐵通道地圖中的自動販賣機、垃圾桶、掩體箱就像空氣一樣，玩家和敵人可以直接穿透它們。而且在「空間扭曲」事件中，當通道變彎時，碰撞邊界與視覺圖像脫節。
*   **背後的邏輯病因**：這些裝飾物的座標與大小沒有被登記到「物理碰撞地圖」中。更困難的是，當空間因為扭曲變成 S 型時，這些裝飾物的視覺模型被彎曲公式移位了，但如果碰撞邊界沒有跟著用數學公式計算移位，就會出現「看得到但撞不到，或者撞到隱形東西」的靈異現象。
*   **我是如何解決的**：
    - 建立地鐵專用的碰撞清單（`FACILITY_COLLIDERS`），將販賣機等物體登記在案。
    - 當空間扭曲事件發生時，在每一幀的碰撞判定中，將這些障礙物的座標同步套用 Sine 正弦彎曲公式進行數學投影變形，確保物理碰撞界限跟著視覺模型一起彎曲移位。

### 14. 管理員跳關後留在原地問題 (Admin Skip Wave Teleportation Bug)
*   **遊戲中的現象**：管理員使用作弊鍵「跳過波次」後，雖然順利開啟了通道門，但玩家仍停留在起點，必須自己跑 240 公尺才能走到下一關，測試起來很沒效率。
*   **背後的邏輯病因**：跳波次的邏輯只清空了敵人的生命值並開啟了閘門，但忘了「傳送玩家」。
*   **我是如何解決的**：在跳波次的邏輯中加上地理位置傳送指令：一旦在地鐵通道中使用跳過波次，自動將玩家的 3D 相機座標 Z 軸傳送到通道盡頭的撤離門附近（`z = -105` 米），免去無意義的跑圖時間。

### 15. 個人倉庫物品圖片背景與尺寸問題 (Stash Item Image Fit Bug)
*   **遊戲中的現象**：個人倉庫裡的武器（例如 AK47）和醫療包有黑底或白底的背景色，且拖曳到格子裡時，武器圖片沒有填滿它所佔據的格子，看起來非常凌亂且廉價。
*   **背後的邏輯病因**：圖片沒有進行去背處理，且 CSS 樣式中圖片的長寬比例被寫死了，沒有隨著網格系統動態拉伸。
*   **我是如何解決的**：
    - 將所有物品圖片處理為去背的 PNG 檔。
    - 修改 CSS 佈局，讓圖片的寬高能 100% 隨物品佔用的網格格數（例如武器佔用 4x2 個格子，醫療包佔用 1x1 個格子）進行自適應拉伸，使排版整齊貼合。

### 16. 點開網頁黑屏/未定義變數崩潰問題 (Weapon SelectedMap ReferenceError Bug)
*   **遊戲中的現象**：剛點開遊戲網頁，畫面一片漆黑，完全沒有加載主選單，按下鍵盤也沒反應，遊戲直接當掉。
*   **背後的邏輯病因**：這在程式中稱為 **`ReferenceError` (未定義變數錯誤)**。因為在設計「電力中斷事件中，玩家主武器會自動開啟手電筒」的功能時，武器元件（`Weapon`）需要知道目前是什麼地圖（`selectedMap`）。但我們忘了在傳遞資料時，把這個地圖變數放進 `Weapon` 的參數清單中。React 在編譯時找不到這個變數，就判定為致命錯誤，直接讓整個網頁崩潰。
*   **我是如何解決的**：進行「狀態提升（State Hoisting）」。將手電筒所需的控制參數和 Reference 提升到最頂層的 [App.jsx](file:///d:/projects/delta%203D/src/App.jsx) 中，再統一作為 Props 傳給底下的每一個元件，消除了未定義變數的崩潰問題。

### 17. 敵軍 AI 位移碰撞未定義 activeColliders 崩潰問題 (Enemy ActiveColliders ReferenceError Bug)
*   **遊戲中的現象**：進入遊戲關卡後，畫面瞬間凍結，控制台瘋狂跳出 `ReferenceError: activeColliders is not defined` 報錯。
*   **背後的邏輯病因**：類似於 Bug 16。我們在編寫敵軍 AI 的碰撞阻擋邏輯時，直接使用了「碰撞體清單（`activeColliders`）」這個變數。然而，敵軍元件（`Enemy`）在自己內部的函數中並沒有定義這個變數，也沒有從外部傳進來。Three.js 引擎在每秒 60 幀的渲染循環中，一讀到這一行就因為找不到變數而崩潰，導致畫面卡死。
*   **我是如何解決的**：在敵軍元件的每幀渲染函數（`useFrame`）頂部，動態獲取當前地圖的碰撞體清單，並套用空間扭曲公式計算變形後的坐標，確保敵人在每幀移動時都能找到對應的變數進行判定。
