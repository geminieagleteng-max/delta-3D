## 📋 歷史 Bug 紀錄與防範清單 (Checklist)

### 1. 敵軍 AI 出生後集體消失問題 (Enemies Disappear Bug)
*   **Bug 症狀**：敵軍生成後會直接往地圖中心點 `(0, 0, 0)` 移動，全部重疊縮在中心點的軍規木箱堆中，從玩家視角看起來像憑空消失了。
*   **原因分析**：地面敵軍 AI 的尋路初始化 `useEffect` 中，依賴了 `meshRef.current` 作為判定。但在 React 首輪掛載 (mount) 時 `meshRef.current` 尚為 `null`，導致該 Effect 塊被跳過，敵軍的 `currentTarget` 未能成功初始化而保持預設的 `(0, 0, 0)`。
*   **修復方案**：在初始化 Effect 中移除對 `meshRef.current` 的依賴，直接讀取 `data.position`（出生點座標）計算距離自身最近的掩體。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 敵軍生成後，是否會朝自身附近的掩體（碉堡、沙包、貨櫃等）前進，而不會全部往 `(0, 0, 0)` 移動？
    - [ ] 在 `Enemy` 元件的 `useFrame` 中，是否設有邊界限幅 (`[-118, 118]`)，防範敵人越界出牆？

### 2. 隱形空氣牆限制問題 (Invisible Air Walls Bug)
*   **Bug 症狀**：雖然地圖外圍圍牆已擴建至 $\pm 120$ 米，但玩家走到地圖的一半（X/Z 約 $\pm 58$ 米）時，會撞上隱形的牆壁無法前進。
*   **原因分析**：在 `PlayerController` 內部的物理移動邊界限幅常數 `mapLimit` 仍被舊程式碼硬編碼為 `58` 米，與地圖網格尺寸不同步。
*   **修復方案**：將 `PlayerController` 內的 `mapLimit` 修正為 **`118`** 米，以完美契合 $\pm 120$ 米的圍牆邊界。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 玩家是否能一路狂奔至地圖邊緣的水泥大高牆（$\pm 118$ 米處），且在 $\pm 58$ 米處無 any 阻礙？
    - [ ] 每次調整地圖外圍大牆（`PerimeterWalls`）時，是否同步調整 `PlayerController` 中的 `mapLimit` 與手榴彈的邊界？

### 3. 掩體沒有碰撞體積問題 (No Cover Collisions Bug)
*   **Bug 症狀**：玩家可以直接「靈體穿牆」走入安全碉堡的沙包牆、中場軍規集裝箱、混凝土牆壁、木箱以及物資補給站內部。
*   **原因分析**：地圖上的防禦工事僅是視覺 3D 網格，在玩家的運動物理中沒有對其進行任何位置阻擋判定。
*   **修復方案**：
    - 在 `PlayerController` 頂部宣告一個包含全場景掩體 2D 座標與尺寸的靜態 **`STATIC_COLLIDERS`** 陣列。
    - 將玩家位移拆分為獨立的 X 軸與 Z 軸移動步驟。每次位移後，計算玩家圓形邊界（半徑 `0.45`）與所有碰撞箱的 AABB 相交。
    - 若偵測到碰撞，即時將玩家推回最近的碰撞箱邊緣外，實現平滑的**滑動碰撞 (Sliding Collision)**。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 玩家朝著碉堡沙包牆、貨櫃、混凝土壁、木箱和補給站奔跑時，是否會被擋住，無法再穿透模型？
    - [ ] 斜向貼牆奔跑時，是否能流暢地沿著掩體邊緣滑行前進，而不會被卡死或震動？
    - [ ] 後續地圖若新增任何固體裝飾或障礙物，是否已手動將其座標與 AABB 尺寸計入 `STATIC_COLLIDERS` 陣列中？

### 4. 裝彈中途被意外取消問題 (Reload Instantly Cancelled Bug)
*   **Bug 症狀**：按下 R 鍵時，換彈進度條（RELOADING...）一閃即逝，子彈完全沒有被補滿。
*   **原因分析**：監聽 R 鍵的 `useEffect` 將 `isReloading` 放入了依賴項陣列。當玩家按 R 開始裝彈時，狀態更新為 `isReloading = true`。這立即觸發該 Effect 進行 Cleanup 卸載，而原 Cleanup 函式寫有 `clearTimeout(reloadTimeoutRef.current)`，這導致剛啟動的換彈計時器在 $1$ 毫秒內被直接抹除，玩家永遠無法完成換彈。
*   **修復方案**：移除了 R 鍵事件監聽 Cleanup 內部的 `clearTimeout`，改為宣告一個單獨無依賴項的 `useEffect` 在整個 `App` 元件被卸載時才做定時器清除。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 當子彈消耗後按下 R 鍵，是否會顯示完整換彈進度條（步槍 1.5 秒 / 手槍 1.0 秒），並在完成後正確補滿彈藥？
    - [ ] 裝彈期間，快速切換武器（數字鍵 1 或 2），裝彈是否被正確中斷？切回原武器時，子彈數是否維持未補滿狀態？

### 5. 手槍不能開槍/扣除步槍彈藥問題 (Pistol Firing Stale Closure Bug)
*   **Bug 症狀**：切換至副武器 M9 手槍後，點擊開火在 HUD 上沒有任何彈藥扣除反應（看起來像手槍壞了），但隨後切回步槍卻發現步槍子彈變少了；一旦步槍彈藥歸零，手槍也跟著完全無法開槍。
*   **原因分析**：在 `PlayerController` 中，綁定滑鼠點擊監聽的 `useEffect` 依賴項為 `[]`（為了效能與不重複註冊）。這導致 `handleMouseDown` 產生了 React **過期閉包 (Stale Closure)** 效應，鎖定在初次掛載渲染時的 `fireOneBullet` 作用域中。此時該作用域捕獲的 `setAmmo` 為步槍的 `setPrimaryAmmo`。手槍是半自動單發，點擊開火完全依賴 `handleMouseDown`，因此每開一槍都在扣步槍子彈。
*   **修復方案**：在 `PlayerController` 中，使用 Refs 緩衝對外傳入的 Derived 狀態及更新回呼函式（`setAmmoRef`、`enemiesRef`、`onHitEnemyRef`、`addImpactEffectRef`、`addCasingRef`），並透過副作用實時更新。在 `fireOneBullet` 中改以 `.current` 讀取最新狀態，擺脫過期閉包捕獲。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 切換到副武器 M9 手槍後，射擊時右下角 HUD 的手槍彈藥量是否會正確扣減（每次 1 發，上限 15 發）？
    - [ ] 手槍開火射擊時，步槍的彈藥（上限 30 發）是否維持不變？
    - [ ] 在 `PlayerController` 的事件監聽器中新增任何存取 React State 或 Props 的邏輯時，是否都已透過 `useRef` 包裝，避免受到 `[]` 依賴產生的過期閉包影響？

### 6. 敵軍子彈穿透掩體打中玩家問題 (Bullets Penetrate Covers Bug)
*   **Bug 症狀**：當玩家隱藏在厚實的沙包或高大集裝箱後方時，遠處的敵軍依然能穿透障礙物開槍打中玩家扣血，發射出的紅色雷射線直接穿過實體模型。
*   **原因分析**：敵軍的開火邏輯僅判定了距離，但在發射子彈時沒有對場景進行任何射線碰撞與視線遮擋檢測。
*   **修復方案**：在 `Enemy` 的開火判定中，加入 `THREE.Raycaster` 視線遮擋檢查（Line of Sight Raycast）。射線起點設定為敵軍頭部，終點為玩家相機位置。若途中最接近的碰撞物體是沙包、貨櫃或混凝土牆，即判定視線受阻，敵軍將停止對玩家扣血與發射子彈。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 當玩家躲藏在集裝箱、混凝土防撞牆或起點碉堡的沙包牆正後方時，遠處的敵軍是否會停止朝玩家開火？是否有任何紅色彈道雷射穿過掩體模型？

### 7. 行動端 HUD 面板遮擋與按鍵重疊問題 (Mobile HUD Overlap Bug)
*   **Bug 症狀**：左下角戰術背包 HUD 面板遮擋虛擬搖桿；底部狀態卡片寬度過寬，遮擋左右側按鍵；右上角暫停鍵與任務資訊面板重疊；右下角開火動作按鈕太貼底，被系統底部導航欄擠出螢幕。
*   **原因分析**：行動端視窗尺寸較小，PC 端的 HUD 配置未針對窄高/矮寬螢幕進行排版分流，造成元素相互疊加或超出安全區域（Safe Area）。
*   **修復方案**：
    - 在 `@media (max-width: 1024px) or (max-height: 550px)` 中重構排版：
    - 背包面板 `.hud-backpack-panel` 移到左上角小地圖正下方 (`top: 110px; left: 15px; transform-origin: top left`)。
    - 底部卡片 `.hud-bottom` 改為水平置中對齊底部，並將單張卡片寬度由 `220px` 壓縮至 `108px`，緊湊排列避開搖桿與按鈕。
    - 暫停鍵 `.mobile-pause-btn` 移至雷達右側 (`left: 195px; top: 12px;`)；切槍鍵 `.btn-switch` 移至右上角 (`right: 15px; top: 12px;`) 躲開任務欄。
    - 右下動作按鍵容器與開火按鈕底邊上抬 `10px`，防止貼底。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 模擬行動裝置橫屏時，左下角虛擬搖桿方圓 150px 內是否沒有任何背包面板遮擋？
    - [ ] 底部血量、彈藥等卡片是否完美置中，不與左右操作按鍵交疊？
    - [ ] 右下角開火、跳躍、裝彈按鈕是否能完整顯示，不被手機底部橫條手勢截斷？

### 8. 行動端視角旋轉滑動靈敏度過低問題 (Mobile Camera Rotation Insensitive)
*   **Bug 症狀**：在手機或平板端滑動螢幕右側空白區域試圖旋轉玩家視角時，鏡頭轉動極為吃力與費勁，需要滑動多次才能完成轉向。
*   **原因分析**：`PlayerController` 在 touchmove 事件中，觸控點位移 delta 計算完畢後乘以的 `sensitivity` 參數僅為 `0.0035`，這在手機上被嚴重稀釋，導致轉向非常不靈敏。
*   **修復方案**：在 `PlayerController` 的 touchmove 事件處理器中，將靈敏度參數從 `0.0035` 提高一倍至 **`0.007`**，使行動端滑動轉向更為輕盈靈活。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 在行動端右側空白區滑動手指，視角旋轉是否反應迅速，且旋轉速度相比之前更為靈活流暢？

### 9. 管理員帳號權限硬編碼與本機資料不同步問題 (Admin Lack of Persistence)
*   **Bug 症狀**：先前將管理員權限以硬編碼寫在註冊與解析層，只要註冊名為 `"distant star"` 的使用者皆能自動升格，這不安全且如果使用者沒重登或沒刷新就無法即時生效。
*   **原因分析**：管理員屬性 `isAdmin` 與金幣值 `coins` 在資料庫 JSON 記錄中並未被真實保存，而是由前端代碼動態攔截並強行改寫。這導致資料庫屬性與前端代碼產生邏輯衝突，且無法彈性分配管理員。
*   **修復方案**：
    - 移除了程式碼在註冊與解析使用者時的 `distant star` 動態 runtime 攔截代碼。
    - 直接在雲端資料庫（KVdb）中，將 `"distant star"`（暱稱 `"遙星"`) 帳號本身的屬性修改為 `"isAdmin": true` 與 `"coins": 99999999`。
    - 在本機快取 [getAccounts](file:///d:/projects/delta%203D/src/utils/account.js#L232) 載入時，針對 `"distant star"`、`"遙星"`（帳號名或暱稱）執行一次性屬性升級遷移並寫回本機 LocalStorage。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 登入 `"distant star"` 帳號後，檢查其 LocalStorage 記錄與雲端記錄，是否已在 JSON 物件中包含 `"isAdmin": true` 和 `"coins": 99999999` 屬性？
    - [ ] 註冊新帳號時，除非資料庫記錄本身被標記，否則是否都不會自動獲取管理員特權？

### 10. 管理員權限不當暴露與缺乏雙重驗證問題 (Admin Privilege Exposure & 2FA)
*   **Bug 症狀**：任何人均可直接利用一般登入框，透過 `"distant star"` 使用者名稱來嘗試登入並碰撞管理員帳號，存在權限暴露與被暴力的安全風險。
*   **原因分析**：管理員帳號沒有獨立的高強度登入渠道與雙重因子（2FA）保護機制。
*   **修復方案**：
    - 限制一般登入框對 `"distant star"` 的驗證登入，自動拋出跳轉至金鑰登入的錯誤提示。
    - 在主畫面上設計獨立的 **【金鑰登入 KEY】** 渠道，要求輸入正確的遊戲金鑰 `DELTA-ADMIN-9999-STAR` (不分大小寫)。
    - 設計基於確定性時間序列的 6 位數 **「電子密碼安全器 (SECURE TOKEN)」**，實作前後 30 秒容差的時間同步動態密碼。
    - 玩家必須輸入 **金鑰 + 密碼 + 動態驗證碼** 才能成功認證登入管理員帳號。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 使用常規登入框登入 `distant star`，是否會被正確阻擋並提示？
    - [ ] 金鑰登入時，金鑰不符、密碼不符或驗證碼不符，是否均能正確顯示對應錯誤，且拒絕進入大廳？
    - [ ] 在 30 秒區間臨界點（倒數 1~2 秒），輸入當前或下一個驗證碼是否皆可順暢登入？

### 11. 敵軍 AI 撞牆/撞障礙物卡死不動問題 (Enemy AI Stuck on Walls Bug)
*   **Bug 症狀**：在實戰中，部分地面敵軍（如突擊兵、擲彈兵）在移動尋找掩體或衝鋒玩家時，若途中撞上牆壁或掩體，會一直推擠牆面並永遠停留在原地不動，失去威脅性。
*   **原因分析**：敵軍的尋路僅是直線朝著 `currentTarget` (掩體座標) 或玩家座標前進。當他們撞牆時，物理碰撞檢測會將他們推回，導致其實際位移為零。因為實際距離未縮短，敵軍會被卡在 `movingToCover` 或 `rushing` 狀態中無法自拔。
*   **修復方案**：
    - 在 `Enemy` 元件的 `useFrame` 中引進**卡牆偵測機制**（比對移動前後的 Vector3 距離）。
    - 當偵測到敵軍處於移動狀態且每秒移動距離低於預期速度的 15% 時，累計卡牆時間。
    - 若累計卡牆超過 `1.2` 秒，即判定為被卡住，並強制為其**重劃路徑**（切換至衝鋒狀態，或在 `COVERS` 中重新挑選另一個隨機掩體作為目標），藉此引導其轉向繞開障礙物。
*   **🛡️ 預防檢修清單 (Checklist)**：
    - [ ] 敵軍生成後，若途中被木箱、沙包或混凝土牆阻擋，是否會在短時間（1.5 秒內）自動轉向、繞開該障礙物？
    - [ ] 觀察衝鋒兵衝向玩家時若被碉堡擋住，是否會自動改道或尋找別的路線，而不會卡死在牆壁前？

## 🔄 未來更新程式確認流程 (Development Workflow Guidelines)
每次修改 `App.jsx` 或調整場景資產、移動邏輯後，必須執行以下確認程序：

1. **語法與編譯檢查**：
   - 執行 `npm run build`，確保專案無編譯報錯。
2. **新手教學全套測試**：
   - WASD 移動與下蹲（下蹲時視角平滑降至 0.9 米，鬆開回至 1.6 米）。
   - 切換主副武器、換彈（步槍 1.5s / 手槍 1.0s），檢查音效與彈匣落地。
   - 射擊訓練靶，確認命中反饋（普通擊中白色、爆頭擊中紅色並顯示 HEADSHOT）。
   - 投擲手榴彈、進行補給箱互動。
3. **實戰碰撞與邊界測試**：
   - 跑向四周的 `STATIC_COLLIDERS`（貨櫃、沙包等），貼著牆面斜向移動，檢查是否會穿透或異常卡死。
   - 狂奔至地圖邊線（118 米處），確保無中途空氣牆阻擋。
   - 躲在掩體後方，觀察敵軍步兵開火射線是否被阻截。
